# Environment Module - Smart City Simulation Foundation

## Purpose

Provides the **complete virtual smart city infrastructure** - vehicles, tasks, fog nodes, cloud, and the city environment. This is the foundation that all simulation runs on.

## Core Components

### 1. City (city.py)

**What:** Represents Istanbul city with real geospatial data

```
Istanbul Bounding Box:
  Lat: 40.75 - 41.25 (0.5° north-south)
  Lon: 28.75 - 29.75 (1.0° east-west)

Real-world size: ~50km x 50km
```

**Why Istanbul:**

- Major metropolitan area (15+ million people)
- Complex network topology (bridges, congestion)
- Real CARLA dataset available (vehicle traces)
- Realistic mobility patterns for vehicular systems

**How Used:**

- Vehicle trajectories follow real CARLA paths through Istanbul
- Fog nodes positioned at core network points
- Network latency modeled based on geographic distance

### 2. Vehicle (vehicle.py)

**What:** Individual connected vehicle in the city

```
Vehicle State:
  - Position (latitude, longitude, altitude)
  - Speed (dynamically changing)
  - Connected Fog Node (current handoff target)
  - Task Queue (pending YOLOv5 work)
  - Active Task (currently offloading)
```

**Why Needed:**

- Source of computational tasks (10 YOLOv5 detections per second)
- Mobile: Changes fog affiliation as it moves around city
- Realistic QoS pattern: Low bandwidth initially, peaks at crossings

**How Used:**

- Follows CARLA trajectory data (real vehicle paths)
- Generates tasks continuously at 10 Hz
- Handoff to fog node happens 10-30 times per simulation
- Network bandwidth traced from CRAWDAD dataset

### 3. Task (task.py)

**What:** Individual computational job - YOLOv5 inference step

```
Task Structure (DAG):
  YOLOv5 5-step pipeline

  Frame → [PreProc] → [Inference] → [PostProc] → [Send Result]
           preproc    inference    postproc      result_xmit

  Total MI: ~500M instructions
  Deadline: ~380ms (from real YOLOv5 latency benchmarks)
```

**Why 5-step DAG:**

- Matches real YOLOv5 architecture
- Each step can execute on different resources
- Some steps can parallelize across fog nodes
- Step 1 can start on vehicle, rest in fog/cloud

**How Used:**

- Generated by vehicles at 10 Hz
- Scheduled by Agent1 (task placement)
- Routed by Agent2 (SDN routing)
- Each step has timestamped arrival/deadline
- Tracked for deadline success metrics

### 4. Fog Node (fog_node.py)

**What:** Edge computing node positioned in Istanbul

```
Fog Node Infrastructure:
  - Compute: 2000 MIPS each
  - Memory: 512 MB buffer for tasks
  - Network: Connected via switches
  - Location: Fixed geographic positions

Default 4 nodes positioned at:
  Beyoglu (central), Fatih (historic),
  Kadikoy (asian side), Bagcilar (airport)
```

**Why Fog (not just cloud):**

- **Latency:** 5-50ms for nearby vehicles vs 30-100ms+ for cloud
- **Bandwidth:** Reduces backhaul traffic to cloud
- **Resilience:** Works even if cloud temporarily unavailable
- **Distributed:** Naturally handles traffic patterns

**How Used:**

- Accepts task submissions from vehicles/other fog
- Executes tasks (compete with MIPS budget)
- Forwards non-critical tasks to cloud
- Collects network latency metrics for learning

### 5. Cloud (cloud.py)

**What:** Central data center with unlimited compute

```
Cloud Server:
  - Compute: 8000 MIPS (4x more than single fog)
  - Network: 30ms base WAN latency + jitter
  - Location: Centralized
  - SLA: Always available
```

**Why Separate:**

- Unbounded compute (suitable for heavy "Boulder" tasks)
- Longer latency (30ms+ network delay)
- Guaranteed reliability vs edge uncertainty
- Cost model: More expensive per task

**How Used:**

- Receives "Boulder" tasks from TOF broker
- Receives overflowed tasks from congested fog
- Provides fallback if fog nodes fail
- Contributes to baseline performance (68%, 81%)

## Real Data Integration

### CARLA Trajectories

```
Source: 50 real vehicle paths through Istanbul
Format: (time, vehicle_id, lat, lon, speed)
Usage: vehicles follow exact traces
Result: Realistic mobility, handoff patterns
```

### YOLOv5 Benchmarks

```
Source: Measured latency of real YOLOv5 execution
Format: Task MI values, execution times by device
Usage: Tasks inherit realistic computational profiles
Result: Accurate resource contention, scheduling challenge
```

### CRAWDAD 4G Traces

```
Source: Real Istanbul 4G network bandwidth/latency
Format: Time-series bandwidth, cellular trace
Usage: Vehicle-to-fog link bandwidth model
Result: Realistic network constraints
```

- Tracks fog node locations and coverage areas
- Coverage radius: 250m per fog node
- `get_fog_in_range()`: finds which fogs cover a vehicle's position

## System Flow

```
Vehicle (in Istanbul)
    ↓ generates YOLOv5 task every 100ms
    ↓
Task created (5 DAG steps, 200ms deadline)
    ↓
TOF-Broker decides: Heavy task? → Cloud : Light task? → Fog+NSGA2
    ↓
Agent1 picks best fog (A, B, C, D) or cloud
    ↓
Execute on selected resource
    ↓
Track latency, energy, deadline success
```

**Bottom line:** These files build the virtual environment where vehicles generate object detection tasks and the system decides where to execute them (fog or cloud).
